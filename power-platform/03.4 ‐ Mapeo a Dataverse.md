> **Subpágina de [[03 Modelo de datos y entidades]].** Este documento especifica el tipo Dataverse de cada atributo del modelo objetivo. **Dataverse es la base estructural principal de datos del MVP**, solicitado por el cliente para mantener consistencia con sus otros módulos. ## Convenciones de tipos
 
| Tipo Dataverse | Cuándo se usa en este modelo |
|---|---|
| **Single line of text** | Códigos, nombres, textos cortos, horas en formato `HH:mm` |
| **Multiline text** | Observaciones, notas y descripciones que pueden ser largas |
| **Whole number** | Contadores, cantidades, segundos — sin decimales |
| **Decimal number** | Kilos, temperaturas, porcentajes, pH — precisión exacta |
| **Date only** | Fechas operacionales sin hora |
| **Date and time** | Timestamps de auditoría |
| **Two options** | Campos sí / no |
| **Choice** | Listas cerradas de valores definidos |
| **Lookup** | Relación (FK) a otra tabla de Dataverse |
 
> **Nota sobre horas:** Dataverse no tiene tipo `TIME` nativo. Todos los campos de hora se almacenan como `Single line of text` en formato `HH:mm`.
 
> **Nota sobre auditoría:** Dataverse provee columnas nativas `Created On` y `Modified On` que se llenan automáticamente. Los campos `created_at` y `updated_at` se implementan con esas columnas nativas — no es necesario crearlas manualmente.
 
> **Nota sobre `rol`:** presente en todas las tablas de proceso como `Single line of text`. El backend determinará los 5 perfiles permitidos y su lógica de validación.

## LotePlanta

**Restricciones de negocio:** `kilos_neto_conformacion` no puede superar `kilos_bruto_conformacion`. El campo `disponibilidad_camara_desverdizado` solo es relevante cuando `requiere_desverdizado = true`; en caso contrario queda en `no_aplica`.

## SequenceCounter (Correlativos)

Tabla auxiliar de backend para la generación automática de códigos operacionales. Gestiona los correlativos de Bin, LotePlanta y Pallet de forma atómica.

**Pendiente de implementación en Dataverse.** En el corte actual, esta tabla existe solo en SQLite. El nombre sugerido en Dataverse es `CaliPro_correlativos`.

| Campo | Tipo Dataverse | Notas |
|---|---|---|
| `CaliPro_entity_name` | Single line of text | `'lote'`, `'bin'`, `'pallet'` |
| `CaliPro_dimension` | Single line of text | Clave de agrupación: `temporada_codigo` para lote; `YYYYMMDD` para bin y pallet |
| `CaliPro_last_value` | Whole number | Último correlativo asignado. El siguiente es `last_value + 1` |
| `created_at` | → columna nativa `Created On` | No crear manualmente |
| `updated_at` | → columna nativa `Modified On` | No crear manualmente |

**Restricción:** `UNIQUE (CaliPro_entity_name, CaliPro_dimension)`.

> La lógica de incremento atómico en Dataverse requiere una estrategia de concurrencia explícita (optimistic concurrency con ETag o tabla de bloqueo). El backend Django es el único responsable de gestionar los correlativos — no deben usarse Power Automate ni fórmulas de Dataverse para esto, para mantener consistencia entre los modos SQLite y Dataverse.

## Consideraciones al crear las tablas en Dataverse
 
**Lookup y campo de visualización.** Dataverse usa GUIDs internamente. Al configurar cada Lookup hay que indicar el campo de visualización — en todos los casos debe ser el código de negocio (`id_bin`, `id_lote_planta`, `id_pallet`).
 
**Unicidad.** Las restricciones de unicidad de `id_bin`, `id_lote_planta` y `id_pallet` se implementan como **Alternate Key**. Para `BinLotePlanta.bin_id`, `PalletLotePlanta.lote_planta_id` e `IngresoAPacking.lote_planta_id`, la unicidad se valida en la capa de aplicación.
 
**Inmutabilidad de códigos.** La inmutabilidad de `id_bin` e `id_pallet` se implementa bloqueando el campo en los formularios una vez guardado, o mediante Power Automate.
 
**Restricciones de rango y consistencia temporal.** Se implementan como **Business Rules** en Dataverse para cada tabla.
 
**Campo `disponibilidad_camara_desverdizado`.** Este campo en `LotePlanta` es nullable en el modelo conceptual (no aplica cuando `requiere_desverdizado = false`). Dado que `Two options` en Dataverse no admite null, se recomienda implementarlo como `Choice` con tres valores: `Disponible`, `No disponible`, `No aplica`. Alternativamente, añadir una Business Rule que lo oculte o bloquee cuando `requiere_desverdizado = No`. La decisión queda a criterio del equipo de implementación.
 
**Campo `rol`.** `Single line of text` en todas las tablas de proceso. El backend validará los 5 perfiles permitidos y la lógica de acceso.
 
**Choice `a_o_r`.** Local Option Set con tres opciones: `Aprobado`, `Objetado`, `Rechazado`.
 
**Campos `Two options`.** Renombrar las etiquetas al nombre del campo para mayor claridad en los formularios.
 
**Deuda técnica pendiente.** Los campos `linea_proceso`, `categoria_calidad` y `proceso` (Desverdizado) son `Single line of text` por ahora. Cuando el cliente entregue los valores definidos, se migran a `Choice`. Los campos de `CalidadDesverdizado` y `CalidadPallet` están propuestos — deben validarse con el cliente.
 
**Nota sobre temporada.** No existe campo `temporada` en ninguna tabla. Las consultas y agrupaciones por temporada se construyen con filtros de rango de fechas sobre `fecha_cosecha` en `Bin`. Esta lógica corresponde al backend o a las vistas y reportes que se construyan sobre el modelo.
